## 1. 总体结论

本次测试的总体结论是 `PASS`，这说明在当前 Ubuntu 环境、当前主板插槽、当前驱动和当前测试参数下，这块 `/dev/nvme1n1` 至少通过了下面几类最基础、最重要的验证：

1. 裸盘写入后可以正确读回，未发现直接的数据一致性错误。
2. `nvme flush` 命令可以正常执行。
3. `blkdiscard` 可以正常执行，discard 后重新写入也没有异常。
4. `fio` 的顺序、随机、混合读写都能正常跑通，没有明显 I/O error。
5. 通过 C 工具下发 NVMe Admin 命令成功，说明低层管理命令路径是通的。
6. 通过 C 的 `O_DIRECT` 校验成功，说明 direct I/O 路径在这次测试条件下也是正常的。
7. SMART 前后没有出现新的介质错误、错误日志增长或关键告警。

如果站在“基础功能验证”的角度，这次结果是比较干净的。

## 3. 功能测试解读

### 3.1 Python 基础写入/回读校验

这项测试在 `16 MiB`、`128 MiB`、`512 MiB` 三个位置各写了 `8 MiB` 数据，并且每个区域都完成了 hash 一致性校验。

这说明：

1. 不是只测了盘头一个点，而是测了多个 LBA 区域。
2. 在这几个区域上，写入数据后再读出，结果与期望完全一致。
3. 至少在“基础裸盘功能正确性”这一层，没有发现直接的数据损坏。

这是整个项目里最基础也最核心的功能测试结果。

### 3.2 Flush 测试

`Flush` 返回成功，且 flush 前后的样本读取没有异常。

这说明：

1. NVMe Flush 命令路径是通的。
2. 这块盘在当前环境下能够响应数据持久化相关命令。

要注意，`Flush PASS` 不等于“断电保护能力已经验证完毕”。它只能说明 flush 命令执行正常，不能替代真正的掉电测试。

### 3.3 TRIM / Discard 测试

这项测试通过，而且 discard 后抽样读回是全 `0`。

这说明：

1. 这块盘在这次测试区域上，对被 discard 的块返回了全 0。
2. 更关键的是，discard 后重新写入新数据并再次校验成功。

这里真正重要的是第二点。因为 SSD 在 discard 之后返回什么值，协议层不一定强制统一；而“discard 后能否继续稳定重写并正确读回”才是更稳妥的工程判断标准。

## 4. 性能测试解读

先给出一个总判断：这次 `fio` 更适合被理解为“性能冒烟测试”，不是完整性能评估。原因是这组任务数据量较小，而且不是长时间稳态测试，所以它主要回答的是“能不能稳定跑通、有没有明显异常”，而不是“这块盘的最终性能定论是什么”。

### 4.1 顺序性能

本次结果大致为：

1. 顺序写 `128k`：约 `1.27 GiB/s`
2. 顺序读 `128k`：约 `2.68 GiB/s`

这个结果说明：

1. 顺序读显著高于顺序写，属于常见现象。
2. 顺序路径整体是健康的，没有出现明显跑不起来或吞吐极低的问题。

如果你以后在面试里讲，可以说这块盘在本次环境下表现出“读优于写”的典型顺序特征。

### 4.2 随机性能

本次结果大致为：

1. 4k 随机写：约 `259k IOPS`，约 `1.01 GiB/s`
2. 4k 随机读：约 `39.5k IOPS`，约 `154 MiB/s`

这里最值得你注意的是：本次随机写明显高于随机读。

这说明什么？

1. 它至少说明设备和 I/O 路径没有挂掉，因为两项都成功完成了。
2. 但它不适合直接被解释为“这块盘随机读天生就只有这么低”。

更稳妥的理解是：

1. 这是一组短时、冒烟性质的结果。
2. 消费级 SSD 的随机写可能会受缓存策略、固件行为、测试区域状态影响。
3. 4k 随机读结果相对偏低，值得你后续再做一次更规范的复测。

建议的复测方式是：

1. 把 `fio` 改成 `time_based=1`
2. 增加 `runtime`
3. 适当加入 `ramp_time`
4. 分别重复跑 3 次，观察波动

所以，当前这组结果的正确表述是：`随机功能路径正常，但 4k 随机读表现相对保守，需要进一步做稳态复测确认。`

### 4.3 混合读写

`randrw_4k_70read` 的结果大致为：

1. 读侧：约 `38.7k IOPS`，约 `151 MiB/s`
2. 写侧：约 `16.6k IOPS`，约 `64.7 MiB/s`

这组结果和前面的随机读写结果是能互相印证的：

1. 混合负载里的读侧表现，和单独的 4k 随机读接近。
2. 混合负载里的写侧表现，明显低于单独随机写，这是正常现象。

原因是混合负载本来就在模拟真实业务场景，读写竞争同一设备资源，写入不可能还保持纯随机写时的最佳状态。

从学习角度，这项结果说明你已经不只是会跑“纯读”“纯写”，而是开始接触更接近真实业务的混合负载模型。

## 5. 延迟解读

这次报告里除了吞吐和 IOPS，还有平均延迟和 P99 延迟。

这里最值得学习的是：`平均值正常` 不等于 `尾延迟一定好看`。

例如：

1. 4k 随机读平均延迟约 `807 us`，但 `P99` 达到约 `8.45 ms`
2. 70/30 混合读写中的读侧平均延迟约 `816 us`，`P99` 约 `8.59 ms`

这说明：

1. 大部分请求完成得还可以。
2. 但尾部仍然存在少数明显更慢的请求。

这在 SSD 测试里很常见，也很重要，因为真实业务体验往往不是被平均值决定，而是被尾延迟决定。
