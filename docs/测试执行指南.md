# 测试执行指南

本文档重点说明：在实际工作中拿到一块全新的 NVMe SSD 后，如何基于本项目开展基础测试。

## 一、测试前准备

### 1. 确认测试对象

你当前指定的目标盘为：

```text
/dev/nvme1n1
```

在真正执行破坏性测试前，建议先手工确认一次：

```bash
lsblk
sudo nvme list
```

确认项：

1. `/dev/nvme1n1` 确实存在。
2. 它不是 Ubuntu 系统盘。
3. 它没有挂载重要分区。
4. 它对应的是你新购买的测试 SSD。

### 2. 安装依赖

Ubuntu 常用安装命令如下：

```bash
sudo apt update
sudo apt install -y nvme-cli smartmontools fio build-essential util-linux
```

这些工具的作用：

1. `nvme-cli`：查询控制器、namespace、SMART、flush 等 NVMe 信息。
2. `smartctl`：读取 SMART/健康日志。
3. `fio`：做性能和 I/O 冒烟测试。
4. `build-essential`：编译 C 工具。
5. `util-linux`：包含 `blkdiscard`、`lsblk` 等常用工具。

### 3. 编译 C 工具

```bash
make -C c_tools
```

编译成功后会生成：

```text
c_tools/nvme_admin_tool
c_tools/nvme_odirect_verify
```

## 二、推荐执行流程

### 方案 A：完整执行

```bash
sudo python3 src/nvme_ssd_test.py \
  --device /dev/nvme1n1 \
  --yes-i-understand-this-will-destroy-data
```

适用场景：

1. 首次完整学习项目
2. 新盘基础验证
3. 需要一份完整测试报告时

### 方案 B：先做功能验证，再做性能

先跳过 `fio`：

```bash
sudo python3 src/nvme_ssd_test.py \
  --device /dev/nvme1n1 \
  --skip-fio \
  --yes-i-understand-this-will-destroy-data
```

确认功能脚本没有问题后，再去掉 `--skip-fio` 执行完整流程。

适用场景：

1. 初次上手，怕一次执行时间过长
2. 先验证裸盘写入、flush、discard 是否正常

### 方案 C：只验证 Python 自动化，不看 C 工具

```bash
sudo python3 src/nvme_ssd_test.py \
  --device /dev/nvme1n1 \
  --skip-c-tool \
  --yes-i-understand-this-will-destroy-data
```

适用场景：

1. 先专注学习 Python 自动化
2. 还没编译 C 工具

## 三、测试结果放在哪里

每次执行都会生成一个独立目录，例如：

```text
reports/run_20260314_153000/
```

目录内主要文件：

1. `report.md`：最终报告，适合直接阅读。
2. `summary.json`：测试项状态汇总，适合后续二次处理。
3. `command_log.json`：记录外部命令、返回码、耗时、输出摘要。
4. `before_*` / `after_*`：测试前后采集到的原始设备信息。
5. `fio_*.json`：每个 `fio` 任务的原始输出。
6. `c_odirect_verify.json`：C 语言 `O_DIRECT` 校验工具的原始输出。

## 四、如何判定是否通过

### 1. Python 基础写入/回读校验

通过标准：

1. 所有测试区都能成功写入。
2. 所有测试区都能按相同 seed 重新生成期望数据并完成逐块比对。
3. 报告中的状态为 `PASS`。

失败表现：

1. 某个 offset 区域回读不一致。
2. 报告中出现 `mismatch_relative_offset`。
3. 说明存在数据一致性风险。

### 2. Flush 测试

通过标准：

1. 写入校验通过。
2. `nvme flush` 命令返回成功。
3. flush 后再次读取样本数据没有异常。

失败表现：

1. `nvme flush` 返回码非 0。
2. 写后回读不一致。

### 3. TRIM / Discard 测试

通过标准：

1. `blkdiscard` 成功执行。
2. discard 后重新写入新 pattern，并再次回读校验通过。

注意：

1. 不要求 discard 后读回全 0。
2. SSD/NVMe 协议下，deallocate 后的返回数据可能是不确定值。

### 4. fio 性能冒烟测试

通过标准：

1. 顺序、随机、混合读写任务都正常执行完成。
2. 没有 I/O error、device busy、permission denied 等错误。
3. 单向负载能看到对应方向的 IOPS、带宽、延迟指标。
4. 混合读写负载能同时看到读侧和写侧指标。

这项测试的重点：

1. 先确认“能稳定跑起来”。
2. 再看性能是否大致符合这块 SSD 的定位。

### 5. C 语言 `O_DIRECT` 校验测试

通过标准：

1. 工具能够以 `O_DIRECT | O_SYNC` 打开目标块设备。
2. 写入阶段无对齐错误、无 I/O error。
3. 回读阶段按相同 seed 重新生成 pattern 后比对一致。
4. 报告中 `match` 为 `true`。

失败表现：

1. `failed to open ... with O_DIRECT`
2. `offset/length/block_size must align to logical block size`
3. `pwrite`、`pread`、`fsync` 失败
4. 报告中出现 `mismatch_offset`

### 6. SMART 对比检查

重点关注：

1. `critical_warning` 是否变为非 0。
2. `media_errors` 是否增加。
3. `num_err_log_entries` 是否增加。

判定建议：

1. `critical_warning != 0`：优先排查，通常应视为高风险。
2. `media_errors` 增加：应视为失败或强风险项。
3. `num_err_log_entries` 小幅增加：不一定立刻判死，但要结合测试现场和错误日志进一步分析。

## 五、测试参数怎么设置

默认参数在：

```text
configs/default_test_plan.json
```

### 1. `write_verify`

```json
{
  "offsets_mb": [16, 128, 512],
  "region_length_mb": 8,
  "chunk_size_kb": 1024,
  "seed_base": 20260314
}
```

参数理解：

1. `offsets_mb`：测试区域起始偏移，避免所有测试都集中在 LBA 0 附近。
2. `region_length_mb`：每个区域写多大。8 MiB 足够做基础一致性验证，又不会拖太久。
3. `chunk_size_kb`：单次写入块大小。这里用 1 MiB，便于观察写入过程。
4. `seed_base`：固定随机种子，保证 pattern 可复现。

### 2. `flush_test`

```json
{
  "offset_mb": 1024,
  "length_mb": 4,
  "chunk_size_kb": 1024,
  "seed": 41001
}
```

设计考虑：

1. Flush 本身不是性能压测，重点是命令语义和数据一致性。
2. 4 MiB 足够小，执行快，也足够验证基本行为。

### 3. `trim_test`

```json
{
  "offset_mb": 1536,
  "length_mb": 16,
  "chunk_size_kb": 1024,
  "write_seed": 52001,
  "rewrite_seed": 52002
}
```

设计考虑：

1. 先写旧数据，再 discard，再重写新数据。
2. 这样既验证 discard 命令成功，也验证后续可正常重新编程。

### 4. `fio_jobs`

顺序读写使用 `128k`，随机读写和混合读写使用 `4k`，这是工作里最常见的入门组合。

参数思路：

1. `rw=write/read/randwrite/randread`：分别对应四种基础负载。
2. `rw=randrw`：表示随机混合读写。
3. `rwmixread=70`：表示读占 70%，写占 30%，这是非常主流的工作负载表达方式。
4. `bs=128k`：更适合顺序吞吐观察。
5. `bs=4k`：更适合随机 IOPS 和混合读写延迟观察。
6. `iodepth=32`：足够形成一定队列压力，但不至于太激进。
7. `numjobs=1`：先保持简单，便于学习。

一个典型混合读写任务如下：

```json
{
  "name": "randrw_4k_70read",
  "rw": "randrw",
  "bs": "4k",
  "iodepth": 32,
  "numjobs": 1,
  "size_mb": 256,
  "offset_mb": 5120,
  "rwmixread": 70
}
```

### 5. `c_odirect_test`

```json
{
  "binary": "./c_tools/nvme_odirect_verify",
  "offset_mb": 5632,
  "length_mb": 8,
  "block_size_kb": 4,
  "seed": 63001
}
```

参数理解：

1. `binary`：C 语言 `O_DIRECT` 校验工具路径。
2. `offset_mb`：校验区域起始偏移，避免和前面 Python/FIO 测试区重叠。
3. `length_mb`：校验长度。
4. `block_size_kb`：单次 `O_DIRECT` I/O 粒度，默认 `4k`，更贴近典型逻辑块大小。
5. `seed`：固定随机种子，保证 pattern 可复现。

设计考虑：

1. `O_DIRECT` 要求对齐严格，`4k` 是最常见、最容易对齐的实验参数。
2. 长度不需要太大，重点是验证“绕过页缓存的直通 I/O 路径是否正确”。

## 六、测试失败时怎么分析

### 1. 先看报告结论

打开：

```text
reports/run_xxx/report.md
```

找到失败项，确认是功能失败、命令失败还是性能失败。

### 2. 再看原始输出

重点文件：

1. `command_log.json`
2. `before_smartctl.json`
3. `after_smartctl.json`
4. `fio_xxx.json`
5. `c_odirect_verify.json`

### 3. 典型排查思路

如果是写读不一致：

1. 检查是否误选了错误设备。
2. 检查设备是否被其他程序同时占用。
3. 复跑同一项，确认问题是否稳定复现。
4. 查看 SMART 和 error log 是否出现增长。

如果是 `fio` 失败：

1. 看是否权限不足。
2. 看设备是否 busy。
3. 看 offset/size 是否越界。
4. 如果是混合读写，单独看读侧和写侧指标，确认问题是在读、写还是总队列层面。

如果是 `O_DIRECT` 校验失败：

1. 先看是否为对齐错误。
2. 再看块大小是否与逻辑块大小兼容。
3. 检查是否有其他程序同时占用目标盘。
4. 对照 `mismatch_offset` 做定点复测。

如果是 SMART 增加：

1. 查看 `media_errors` 和 `num_err_log_entries`。
2. 配合 C 工具查询 error log。
3. 必要时保留现场并停止继续压测。
