# NVMe 基础与工具说明

本文档不是协议全本，而是为了配合本项目，帮助你建立“做测试时真正会用到”的知识框架。

## 一、什么是 NVMe SSD

NVMe 是面向非易失性存储器的主机控制器接口规范。对你这个项目来说，最重要的不是背定义，而是理解它和 SATA SSD 的差异：

1. NVMe 走 PCIe，总体带宽和并行能力更强。
2. 它天然支持多队列，更适合高并发 I/O。
3. Linux 下常见设备节点形态是 `/dev/nvmeX` 和 `/dev/nvmeXnY`。

示例：

1. `/dev/nvme1`：控制器节点
2. `/dev/nvme1n1`：第 1 个 namespace

## 二、控制器和 Namespace 的区别

### 控制器

控制器可以理解成 SSD 的控制单元视角，很多管理命令都从这里下发。

常见信息：

1. 型号
2. 序列号
3. 固件版本
4. 厂商标识

### Namespace

namespace 更偏逻辑存储空间视角，类似“这个控制器暴露出的一个逻辑盘”。

常见信息：

1. 容量
2. 使用情况
3. LBA format
4. namespace id

在本项目中：

1. Python 脚本主要针对 `/dev/nvme1n1` 做裸盘 I/O。
2. C 工具主要针对 `/dev/nvme1` 下发 admin 命令。

## 三、为什么要关心 LBA Format

存储测试里经常会遇到逻辑块大小概念。LBA format 决定了：

1. 一个逻辑块是多少字节
2. 某些 I/O 对齐要求如何满足
3. 性能和兼容性测试中应该怎样设定块大小

常见逻辑块大小是 `512B` 或 `4096B`。如果你以后做更深入的测试，例如对齐写、边界写、跨页写，LBA format 就会变得更加重要。

## 四、为什么 SMART 很关键

SMART 不是“可有可无的附加信息”，而是存储测试里非常重要的一条观察线。

你可以把它理解成设备健康状态和异常线索的摘要。

本项目重点看这些字段：

1. `critical_warning`
2. `temperature`
3. `available_spare`
4. `percentage_used`
5. `media_errors`
6. `num_err_log_entries`

### 1. `critical_warning`

如果这个值非 0，一般说明已经有比较严重的风险信号，例如备用空间不足等。

### 2. `temperature`

温度过高不一定马上报错，但它会显著影响稳定性判断。工作中经常会结合不同负载前后的温升去观察。

### 3. `available_spare`

可以理解为设备内部预留健康度的一种体现，低于阈值通常需要重点关注。

### 4. `percentage_used`

常用于观察设备寿命消耗程度。对新盘来说，这个值通常应比较低。

### 5. `media_errors`

非常关键。它更接近介质层面的错误信号。如果这个值增长，要高度重视。

### 6. `num_err_log_entries`

表示错误日志条目数量。如果它增长了，下一步通常就要去查 Error Log。

## 五、为什么 `fio` 是存储测试常用工具

因为它足够灵活，几乎可以构造绝大多数基础 I/O 负载。

你在本项目里先掌握这几个维度就够了：

1. `rw`：读写模式
2. `bs`：块大小
3. `iodepth`：队列深度
4. `numjobs`：并发任务数
5. `size`：测试数据量
6. `offset`：测试起始偏移
7. `direct=1`：尽量绕过页缓存，贴近块设备真实 I/O
8. `rwmixread`：混合读写场景里读请求所占比例

### 常见组合

顺序读写：

```bash
fio --rw=read --bs=128k
fio --rw=write --bs=128k
```

随机读写：

```bash
fio --rw=randread --bs=4k
fio --rw=randwrite --bs=4k
```

混合读写：

```bash
fio --rw=randrw --bs=4k --rwmixread=70
```

这里的含义不是“跑一个平均值就够了”，而是要同时观察：

1. 读侧吞吐、IOPS、延迟
2. 写侧吞吐、IOPS、延迟
3. 混合比例变化对设备行为的影响

## 六、什么是 `O_DIRECT`，为什么存储测试里经常会提到它

`O_DIRECT` 是 Linux 下打开文件或块设备时常见的一个 flag。它的核心作用是尽量绕过页缓存，让 I/O 更直接地到达块层。

### 为什么它有价值

1. 可以减少页缓存对测试结果的干扰。
2. 更适合做裸盘一致性验证和某些性能验证。
3. 有助于观察更接近设备路径本身的行为。

### 为什么它不适合无脑使用

因为它通常要求：

1. 缓冲区地址对齐
2. I/O 长度对齐
3. 文件偏移对齐

这也是为什么本项目专门用 C 写了 `O_DIRECT` 校验工具，而不是简单在 Python 里硬拼一个版本。

## 七、为什么 Python 和 C 要一起用

### Python 的优势

1. 编排流程快
2. 调命令方便
3. 结果汇总和报告生成效率高
4. 非常适合自动化测试框架

### C 的优势

1. 更容易贴近底层接口
2. 可以直接使用 `ioctl`
3. 更适合做协议层、管理命令层、边界条件类工具

这也是为什么很多存储测试岗位同时看重：

1. Python 自动化能力
2. C 语言基础和系统接口理解
