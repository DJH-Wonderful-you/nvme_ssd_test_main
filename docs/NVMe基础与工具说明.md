# NVMe 基础与工具说明

本文档不是协议全本，而是为了配合本项目，帮助你建立“做测试时真正会用到”的知识框架。

## 一、什么是 NVMe SSD

NVMe 是面向非易失性存储器的主机控制器接口规范。对你这个项目来说，最重要的不是背定义，而是理解它和 SATA SSD 的差异：

1. NVMe 走 PCIe，总体带宽和并行能力更强。
2. 它天然支持多队列，更适合高并发 I/O。
3. Linux 下常见设备节点形态是 `/dev/nvmeX` 和 `/dev/nvmeXnY`。

示例：

1. `/dev/nvme1`：控制器节点
2. `/dev/nvme1n1`：第 1 个 namespace

## 二、Controller和 Namespace 的区别

### Controller（控制器）

Controller可以理解成 SSD 的控制单元视角，很多管理命令都从这里下发。

常见信息：

1. 型号
2. 序列号
3. 固件版本
4. 厂商标识

### Namespace（命名空间）

namespace 更偏逻辑存储空间视角，类似“这个控制器暴露出的一个逻辑盘”。

常见信息：

1. 容量
2. 使用情况
3. LBA format
4. namespace id

在本项目中：

1. Python 脚本主要针对 `/dev/nvme1n1` 做裸盘 I/O。
2. C 工具主要针对 `/dev/nvme1` 下发 admin 命令。

## 三、为什么要关心 LBA Format

存储测试里经常会遇到逻辑块大小概念。LBA format 决定了：

1. 一个逻辑块是多少字节
2. 某些 I/O 对齐要求如何满足
3. 性能和兼容性测试中应该怎样设定块大小

常见逻辑块大小是 `512B` 或 `4096B`。如果你以后做更深入的测试，例如对齐写、边界写、跨页写，LBA format 就会变得更加重要。

## 四、为什么 SMART 很关键

SMART 不是“可有可无的附加信息”，而是存储测试里非常重要的一条观察线。

你可以把它理解成设备健康状态和异常线索的摘要。

本项目重点看这些字段：

1. `critical_warning`
2. `temperature`
3. `available_spare`
4. `percentage_used`
5. `media_errors`
6. `num_err_log_entries`

### 1. `critical_warning`

如果这个值非 0，一般说明已经有比较严重的风险信号，例如备用空间不足等。

### 2. `temperature`

温度过高不一定马上报错，但它会显著影响稳定性判断。工作中经常会结合不同负载前后的温升去观察。

### 3. `available_spare`

可以理解为设备内部预留健康度的一种体现，低于阈值通常需要重点关注。

### 4. `percentage_used`

常用于观察设备寿命消耗程度。对新盘来说，这个值通常应比较低。

### 5. `media_errors`

非常关键。它更接近介质层面的错误信号。如果这个值增长，要高度重视。

### 6. `num_err_log_entries`

表示错误日志条目数量。如果它增长了，下一步通常就要去查 Error Log。

## 五、为什么 `fio` 是存储测试常用工具

因为它足够灵活，几乎可以构造绝大多数基础 I/O 负载。

你在本项目里先掌握这几个维度就够了：

1. `rw`：读写模式
2. `bs`：块大小
3. `iodepth`：队列深度
4. `numjobs`：并发任务数
5. `size`：测试数据量
6. `offset`：测试起始偏移
7. `direct=1`：尽量绕过页缓存，贴近块设备真实 I/O
8. `rwmixread`：混合读写场景里读请求所占比例

### 常见组合

顺序读写：

```bash
fio --rw=read --bs=128k
fio --rw=write --bs=128k
```

随机读写：

```bash
fio --rw=randread --bs=4k
fio --rw=randwrite --bs=4k
```

混合读写：

```bash
fio --rw=randrw --bs=4k --rwmixread=70
```

这里的含义不是“跑一个平均值就够了”，而是要同时观察：

1. 读侧吞吐、IOPS、延迟
2. 写侧吞吐、IOPS、延迟
3. 混合比例变化对设备行为的影响

## 六、什么是 `O_DIRECT`，为什么存储测试里经常会提到它

`O_DIRECT` 是 Linux 下打开文件或块设备时常见的一个 flag。它的核心作用是尽量绕过页缓存，让 I/O 更直接地到达块层。

### 为什么它有价值

1. 可以减少页缓存对测试结果的干扰。
2. 更适合做裸盘一致性验证和某些性能验证。
3. 有助于观察更接近设备路径本身的行为。

### 为什么它不适合无脑使用

因为它通常要求：

1. 缓冲区地址对齐
2. I/O 长度对齐
3. 文件偏移对齐

这也是为什么本项目专门用 C 写了 `O_DIRECT` 校验工具，而不是简单在 Python 里硬拼一个版本。

用户态缓冲区（buffer）的地址、数据偏移量（offset）、以及读取的数据长度，通常都必须是文件系统**逻辑块大小**的整数倍（常见的是512字节或4K）。

用户缓冲区是应用程序在用户态内存中分配的一块区域，用于存放要写入磁盘的数据或从磁盘读取的数据。

**因为 DMA 硬件要求内存地址对齐，而 O_DIRECT 的核心就是让硬件直接访问内存，所以用户缓冲区必须满足硬件的对齐要求。**

## 七、为什么 Python 和 C 要一起用

### Python 的优势

1. 编排流程快
2. 调命令方便
3. 结果汇总和报告生成效率高
4. 非常适合自动化测试框架

### C 的优势

1. 更容易贴近底层接口
2. 可以直接使用 `ioctl`
3. 更适合做协议层、管理命令层、边界条件类工具

这也是为什么很多存储测试岗位同时看重：

1. Python 自动化能力
2. C 语言基础和系统接口理解



## fio详细解读

### 一、核心控制参数（你必须懂的）

这些参数决定了测试的基本形态。

#### 1. `filename`

- **含义**：测试文件或块设备的路径。
- **示例**：`filename=/dev/sdb` 或 `filename=/data/testfile`
- **影响**：
  - 如果直接指定**块设备**（如 `/dev/sdb`），测试的是**裸盘**性能，绕过了文件系统缓存，结果最准确，反映硬件极限。
  - 如果指定**文件系统上的文件**，测试结果包含了文件系统（如 ext4， xfs）的开销（如日志写入、元数据更新）。
  - **警告**：直接写块设备会**破坏文件系统结构**，请务必确认设备无数据。

#### 2. `directory`

- **含义**：设置测试文件存放的目录。fio 会在该目录下创建 `iofile1`， `iofile2`... 等文件。
- **影响**：方便测试特定挂载点的性能。结合 `filename` 使用。

#### 3. `rw` (或 `readwrite`)

- **含义**：I/O 模式。这是最重要的参数之一。
- **可选值及影响**：
  - `read`：顺序读。主要测试磁盘的持续读取速度。对 HDD 影响较大（外圈快），对 NVMe SSD 影响较小（全域高速）。
  - `write`：顺序写。主要测试持续写入速度。会触发缓存算法和垃圾回收。
  - `randread`：随机读。测试寻道/寻址能力。对 HDD 这是致命伤（磁头摆动），对 SSD 这是核心指标（IOPS）。
  - `randwrite`：随机写。**最复杂的场景**。对 SSD 而言，这涉及到写放大、FTL（闪存转换层）映射和垃圾回收。
  - `rw`， `readwrite`：混合顺序读写。
  - `randrw`：混合随机读写。通过 `rwmixread` 参数调整比例（默认 50%读）。

#### 4. `ioengine`

- **含义**：I/O 引擎，即通过什么系统调用来发起 I/O。
- **可选值及影响**：
  - `sync`：基础的 `read/write` 调用。每次 I/O 都陷入内核。
  - `psync`：`pread/pwrite`，基础同步 I/O。
  - `libaio`：**Linux 原生异步 I/O**。这是测试裸盘性能最常用的引擎（配合 `direct=1`）。注意，即使叫异步，如果不配合 `iodepth`，实际上还是同步的。
  - `io_uring`：新一代异步 I/O 引擎，性能更高，减少了用户态和内核态的内存拷贝。适用于最新内核。
  - `mmap`：通过内存映射读写文件。
  - `posixaio`：POSIX 异步 I/O（通常性能不如 libaio）。
  - `rbd`：直接测试 Ceph 块设备。
  - `pvsync`：带有屏障的同步 I/O。

#### 5. `direct`

- **含义**：布尔值（0 或 1）。是否使用 `O_DIRECT` 标志打开文件。
- **影响**：
  - `direct=1`：绕过操作系统 Page Cache。数据直接在用户空间缓冲区与磁盘之间传输。**这是测试真实物理磁盘性能的标准做法**，避免内存充当缓存导致测试数据虚高。
  - `direct=0`：使用 Page Cache。读操作可能直接从内存返回，写操作只是写入内存就返回（脏页后续刷入磁盘）。这测试的是应用程序结合 OS Cache 的整体表现。

#### 6. `bs` (或 `blocksize`)

- **含义**：每次 I/O 请求的块大小。默认 4k。
- **影响**：
  - **小 Block Size (4k， 8k)**：测试**IOPS**（每秒读写次数）。主要考验控制器的处理能力和设备寻址延迟。数据库 OLTP（在线事务处理）场景常用。
  - **大 Block Size (1M， 8M)**：测试**带宽**（Throughput）。主要考验设备的数据传输速率。视频流、大文件拷贝场景常用。
  - 可以使用范围，如 `bs=4k-64k` 来模拟混合大小。

#### 7. `size`

- **含义**：每个 job 线程使用的总数据量。
- **影响**：
  - `size < 设备总容量`：测试局部区域性能。由于 SSD 内部可能有 SLC Cache（模拟单层单元缓存），小范围写入可能全落在高速缓存中，导致测试结果“很好看”，但这不代表稳定态性能。
  - `size = 100%`：测试全盘性能。能反映出盘满时的 GC（垃圾回收）影响。

#### 8. `iodepth`

- **含义**：I/O 深度。在异步引擎（如 libaio，读音lib A-I-O）下，**一次提交给内核的 I/O 请求数量**。
- **影响**：
  - **iodepth=1**：相当于同步队列。系统在处理完一个 I/O 后才发起下一个。
  - **iodepth=高数值 (64， 128)**：让设备可以重排命令，发挥并行能力。对于 NVMe SSD，高队列深度是发挥其性能的关键（考验控制器并发能力）。如果设备支持 NCQ（原生指令队列）或 NVMe 队列，高 iodepth 能提升 IOPS，但会增加延迟。

#### 9. `numjobs`

- **含义**：并行的工作进程（线程）数量。
- **影响**：
  - `numjobs=1`：单进程压测。
  - `numjobs=16`：模拟多线程并发访问。这可以产生比 `iodepth` 更大的压力，但要注意，不同的进程操作不同的文件或同一文件时的锁竞争。
  - **重要**：总并发数 = `numjobs` * `iodepth`。

#### 10. `time_based` & `runtime`

- **含义**：
  - `runtime`：测试持续时间（秒）。
  - `time_based`：如果设置了 `size`，数据写完就停止；加上这个参数，即使数据写完了也继续循环跑，直到时间结束。
- **影响**：对于 SSD，短时间测试无法触发稳态（Steady State），长时间测试才能看到降速（比如 SLC Cache 写爆后掉速）。

### 第一组：同步 vs 异步

这是**处理任务的方式**问题，核心在于 **“发起者需不需要自己盯着结果”**。

#### 1. 同步（Synchronous）

- **类比**：你在柜台排队办业务。你递进去材料，然后**一直站在窗口等着**，直到柜员把处理结果（盖章的文件）递回给你，你才离开。在等待期间，你什么都做不了（阻塞）。
- **技术含义**：
  - 应用程序发起一个系统调用（如 `read`）。
  - 内核开始处理（等待磁盘数据）。
  - **进程（线程）进入休眠状态，让出 CPU**。
  - 数据准备好后，内核唤醒进程，系统调用返回。
- **代码表现**：`read()` 函数返回时，数据已经在 buffer 里了。
- **fio 对应**：`ioengine=sync`，`iodepth` 通常为 1（因为发出一个请求后必须等它完成才能发下一个）。

#### 2. 异步（Asynchronous）

- **类比**：你拿号（取号单），然后**回到座位上玩手机**。当柜员叫到你的号时，你再去窗口拿结果。在这个过程中，你没有一直堵在窗口。
- **技术含义**：
  - 应用程序发起一个系统调用（如 `io_submit` for libaio），告诉内核“我要读这块数据，好了告诉我”。
  - **调用立即返回**，进程（线程）继续运行（可以做其他计算，或者发起更多 I/O）。
  - 内核处理数据。
  - 数据准备好后，内核通过某种方式（信号、回调、或 `io_getevents` 检查）通知应用程序“数据好了，来取吧”。
- **代码表现**：`io_submit` 返回 0（成功提交），稍后调用 `io_getevents` 拿到结果。
- **fio 对应**：`ioengine=libaio` 或 `io_uring`，配合 `iodepth > 1` 使用。

### 第二组：进程 vs 线程

这是**资源分配和调度的单位**问题，核心在于 **“有多少套独立的执行上下文”**。

#### 1. 进程（Process）

- **类比**：独立的公司。
  - 每个公司（进程）有自己的独立办公室（内存空间）、独立的财务（文件描述符表）、独立的员工（线程）。
  - 公司 A 倒闭了，公司 B 完全不受影响。
- **技术含义**：
  - 资源分配的最小单位。
  - 每个进程有独立的虚拟地址空间（4G 虚拟内存）、独立的 PCB（进程控制块）。
  - **切换开销大**（切换页表、刷新 TLB）。
  - **通信复杂**（需要 IPC：管道、共享内存、Socket）。

#### 2. 线程（Thread）

- **类比**：公司里的员工。
  - 所有员工（线程）在同一个办公室（进程的内存空间）里办公。
  - 他们共享打印机、饮水机（共享文件描述符、信号处理等）。
  - 员工 A 把一份文件扔进碎纸机（崩溃），可能会导致整个办公室（进程）陷入混乱。
- **技术含义**：
  - CPU 调度的最小单位。
  - 线程共享进程的地址空间、文件描述符、信号处理器。
  - **切换开销小**（只需保存寄存器、程序计数器）。
  - **通信简单**（直接读写全局变量，但需注意锁）。

### 第三组：I/O 深度（`iodepth`） vs 并行进程/线程（`numjobs`）

这是**压测流量产生方式的区别**，核心在于 **“单条流水线上一次放几个包裹 vs 开几条流水线”**。

#### 1. I/O 深度（`iodepth`）—— 单管的并发

- **类比**：机场安检传送带。
  - 你（单个线程）站在传送带（I/O 队列）前。
  - `iodepth=1`：你放上一个包，包走到尽头被扫描完，你再放下一个包。**传送带上有空隙**。
  - `iodepth=16`：你一次性往传送带上放 16 个包（提交 16 个 I/O 请求）。传送带上满满当当。安检机（磁盘）会尽可能快地处理这些包。
- **技术含义（以 libaio 为例）**：
  - 单线程调用 `io_submit` 一次性提交 N 个请求到内核的 Block Layer。
  - 这 N 个请求可以同时在内核中进行排队、合并、下发。
  - 对于 NVMe 设备，它支持多队列（每个队列深度可达 256 或更高），高 `iodepth` 允许设备的重排序和并行能力发挥到极致。
  - **作用**：让单核 CPU 有能力压满高速设备（如 NVMe SSD）的带宽，**考验设备控制器的并发处理能力和寻道/页缓存命中率**。

#### 2. 并行进程/线程（`numjobs`）—— 多管齐下

- **类比**：机场开设多个安检通道。
  - 你开 8 个安检通道（8 个线程/进程 `numjobs=8`）。
  - 每个通道都有自己的传送带（每个线程有自己的 I/O 上下文），每个传送带上都可以堆满包裹（每个线程有自己的 `iodepth`）。
  - 机场大厅里瞬间涌入了 `8 * iodepth` 个包裹。
- **技术含义**：
  - 每个 `job` 是一个独立的任务（进程或线程）。
  - 如果 `numjobs=8`，`iodepth=16`，则**系统总并发 I/O 数为 128**。
  - 这些 jobs 可能绑定在不同的 CPU 核心上。
  - 它们可能操作同一个文件（产生锁竞争和并发访问）或不同文件。
  - **作用**：增加 CPU 核心的参与度，模拟多应用并发访问的场景，**考验文件系统锁的 scalability（可扩展性）和 CPU 多核处理能力**。当单线程无法压满设备时（例如某些 HDD 或低速介质），可以增加 `numjobs` 来施压。

### 具体关联点：

1. **同步/异步决定了是否需要 `iodepth`**：
   - **同步**：天生 `iodepth` 只能为 1。想要提高并发，只能靠增加 `numjobs`（开多个同步线程）。
   - **异步**：单线程可以通过高 `iodepth` 打出高并发。
2. **进程/线程决定了共享程度**：
   - 如果 `numjobs` 是**线程**（`thread` 参数），它们共享内存地址空间，如果操作同一个文件，会有激烈的锁竞争，可能导致性能比进程模式低（因为锁的开销）。
   - 如果 `numjobs` 是**进程**（默认），各有各的文件表，竞争相对少一些。
3. **总并发 I/O 数的计算公式**：
   - **总 Inflight I/O（飞行中的 I/O 数） = `numjobs` × `iodepth`**
   - 这个总数不能超过操作系统或设备硬件的限制。如果超过，可能会导致请求积压或延迟急剧升高。

## nvme-cli详细解读

`nvme-cli` 是 Linux 环境下管理和诊断 NVMe 固态硬盘的**事实标准工具集**。与 fio 这种**负载生成工具**不同，nvme-cli 是**设备控制与信息采集工具**——它直接通过 IOCTL 命令与 NVMe 驱动程序交互，让用户能够发送标准化的 NVMe 管理命令给固态硬盘的主控。

如果说 fio 问的是“硬盘跑得快不快”，nvme-cli 问的就是“硬盘本身是什么状态、有什么毛病、该怎么设置”。

我将按照功能模块来详细拆解 nvme-cli 的命令体系。

------

### 一、信息查询类命令（最常用）

这类命令用于获取设备的基本信息、健康状态和错误日志。

#### 1. `nvme list`

- **作用**：列出系统中所有 NVMe 设备及其基本信息。
- **输出内容**：设备名称（`/dev/nvme0n1`）、型号、序列号、固件版本、容量、命名空间 ID。
- **使用场景**：刚插上一块新盘，先看看系统认没认出来。

#### 2. `nvme id-ctrl`

- **作用**：读取控制器的**Identify Controller**数据结构（NVMe 协议中最基础的数据结构）。
- **命令**：`nvme id-ctrl /dev/nvme0 -H`
  - `-H` 输出人类可读的解释。
- **查看内容**：
  - 支持的可选特性（如是否支持格式化、是否支持端到端数据保护）。
  - 最大队列深度（`MQES`）、数据传输大小限制。
  - **IEEE 唯一标识符**（全球唯一标识）。
  - **固件版本（FR）**。

#### 3. `nvme id-ns`

- **作用**：读取指定**Namespace**的 Identify 数据结构。
- **命令**：`nvme id-ns /dev/nvme0n1 -H`
- **查看内容**：
  - LBA 格式（扇区大小，通常是 512 或 4096）。
  - 命名空间大小和容量。
  - 是否支持解分配（Discard/Trim）。

#### 4. `nvme smart-log`

- **作用**：**最常用命令之一**。获取设备的 SMART / Health Information 日志。
- **命令**：`nvme smart-log /dev/nvme0`
- **关键指标**：
  - `percentage_used`：**使用寿命**（基于写入量和擦除次数估算的百分比）。这是 SSD 寿命的核心指标，注意不是存储空间占用，而是闪存颗粒的磨损程度。比如显示 15%，意味着预期寿命已消耗 15%。
  - `data_units_written`：**总写入量**（单位是 1000 * 512 字节）。可以据此计算每天的写入量，判断是否符合保修条款。
  - `power_cycles` / `power_on_hours`：通电次数/时间。
  - `unsafe_shutdowns`：异常断电次数。频繁异常断电可能影响数据一致性。
  - `temperature`：当前温度。
  - `critical_warning`：关键警告（如温度过高、备用空间不足、只读模式等）。非零值代表有问题。

#### 5. `nvme error-log`

- **作用**：读取设备的错误日志。
- **命令**：`nvme error-log /dev/nvme0`
- **查看内容**：记录设备报告的过往错误，如校验错误、超时、命令中止等。对于排查硬件问题（如链路不稳定）非常有帮助。

#### 6. `nvme list-ctrl`

- **作用**：查看系统中 NVMe 控制器的路径。

#### 10. `nvme flush`

- **作用**：将控制器缓存中的数据强制写入 NAND 闪存。
- **命令**：`nvme flush /dev/nvme0n1`
- **使用场景**：在安全移除设备前，确保数据全部落盘。
