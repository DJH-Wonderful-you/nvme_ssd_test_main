# 第一点

这些命令都会通过 `subprocess.run()` 直接在 shell 中执行。这些命令都是 **Linux 系统自带的或可独立安装的命令行工具**，每个工具都有特定的用途。让我详细介绍这些命令：

## 1. **blockdev** - 块设备查询工具

bash

```
blockdev --getsize64 /dev/nvme1n1
```



- **来源**：util-linux 包（Linux 基础工具集）

- **功能**：获取块设备大小（字节）

- **类似命令**：

  bash

  ```
  fdisk -l /dev/nvme1n1        # 显示分区信息
  lsblk -b /dev/nvme1n1        # 以字节显示大小
  ```

  

## 2. **lsblk** - 列出块设备信息

bash

```
lsblk -J -O /dev/nvme1n1
```



- **来源**：util-linux 包
- **参数**：
  - `-J`：JSON 格式输出
  - `-O`：输出所有可用列 **Output all**
- **输出示例**：

json

```
{
  "blockdevices": [
    {"name": "nvme1n1", "size": "512110190592", "ro": false, "type": "disk"}
  ]
}
```



## 3. **nvme-cli** - NVMe 设备专用工具集

这是 **最重要的一组命令**，需要单独安装（`apt install nvme-cli`）：

### a) `nvme id-ctrl` - 读取控制器识别信息

bash

```
nvme id-ctrl /dev/nvme1 -o json
```

- **-o**：**output** format（输出格式）
- 指定输出为 JSON 格式
- **功能**：读取 NVMe 控制器的 Identify 数据结构
- **包含信息**：厂商ID、型号、固件版本、NVMe版本、支持的命令集等

### b) `nvme id-ns` - 读取命名空间信息

bash

```
nvme id-ns /dev/nvme1n1 -o json
```



- **功能**：读取特定命名空间的参数
- **包含信息**：容量、LBA格式、块大小、保护信息类型等

### c) `nvme smart-log` - 读取SMART日志

bash

```
nvme smart-log /dev/nvme1n1 -o json
```



- **功能**：读取 NVMe 协议的SMART/健康信息
- **关键字段**：
  - `temperature`：当前温度
  - `percentage_used`：使用寿命百分比
  - `data_units_read/written`：数据读写量
  - `media_errors`：媒体错误计数

### d) `nvme flush` - 刷新缓存到介质

bash

```
nvme flush /dev/nvme1n1 -n 1
```

- **-n**：**namespace** id（命名空间ID）
- 指定要刷新的命名空间是 1

- **功能**：强制将缓存中的数据写入非易失性介质

## 4. **smartctl** - SMART监控工具

bash

```
smartctl -a -j /dev/nvme1n1
```



- **来源**：smartmontools 包（`apt install smartmontools`）
- **参数**：
  - `-a`：显示所有SMART信息 **all** information
  - `-j`：JSON格式输出
- **优势**：比 nvme-cli 提供更全面的厂商特定属性

## 5. **blkdiscard** - 块设备丢弃/TRIM命令

bash

```
blkdiscard -o 1G -l 100M /dev/nvme1n1
```



- **来源**：util-linux 包
- **功能**：对存储设备执行丢弃操作（TRIM）
- **参数**：
  - `-o`：起始偏移量	**offset**
  - `-l`：长度

## 6. **fio** - 灵活I/O测试工具

bash

```
fio --name=test --filename=/dev/nvme1n1 --rw=randread --bs=4k --iodepth=32 ...
```



- **来源**：fio 包（`apt install fio`）
- **功能**：专业的I/O性能测试工具
- **常用测试模式**：
  - `--rw=read`：顺序读
  - `--rw=write`：顺序写
  - `--rw=randread`：随机读
  - `--rw=randwrite`：随机写
  - `--rw=rw`：混合读写

# 第二点

### `os.open(str(self.device), os.O_RDWR | os.O_SYNC)` 详细解释

这行代码的作用是打开一个文件（这里是一个块设备），并返回一个文件描述符 `fd`。

- `str(self.device)`: 这是要打开的设备路径，例如 `"/dev/nvme1n1"`。
- `os.O_RDWR`: 这是一个**打开模式标志**，代表以**可读可写**的方式打开文件。
- `os.O_SYNC`: 这是另一个**标志**，它代表**同步 I/O**。这意味着任何对文件的 `write` 操作都会阻塞，直到数据被真正写入物理硬件并得到确认后才返回，而不是仅仅写入到操作系统的页面缓存中。
- `|` (按位或运算符): 它的作用是将多个标志组合起来。在这里，就是将“可读可写”和“同步写入”两个要求组合在一起，传递给 `os.open` 调用。

**小结**：这行代码以**可读可写且同步**的模式打开了底层的 NVMe 设备，为后续直接、可靠地与硬件交互做准备。

### `os.open`, `os.pread`, `os.pwrite` 的作用

这三个函数是 Python `os` 模块提供的底层系统调用接口，它们绕过了 Python 的文件对象和缓冲，直接与操作系统内核交互。

- **`os.open(path, flags, mode=0o777)`**:
  - **作用**：正如上面解释的，它是打开文件的**底层接口**。它返回一个整数的文件描述符 `fd`，后续的 `read`、`write`、`lseek` 等操作都使用这个 `fd` 来指代打开的文件。
  - **为什么用它**：可以获得对文件打开方式（如同步、直接 I/O）的精细控制，这是标准 `open()` 函数做不到的。
- **`os.pwrite(fd, buffer, offset)`**:
  - **作用**：从指定的 `offset`（偏移量）开始，将 `buffer` 中的数据写入到文件描述符 `fd` 指向的文件中。
  - **关键特性**：这是一个**原子操作**，并且**不会改变文件的当前读写位置**。也就是说，它只在你指定的位置写入数据，不影响其他线程或进程的读写指针。这非常适合多线程环境下的随机写入。
- **`os.pread(fd, n, offset)`**:
  - **作用**：从文件描述符 `fd` 的指定 `offset`（偏移量）开始，读取最多 `n` 个字节的数据。
  - **关键特性**：和 `pwrite` 一样，它也是原子操作且**不会改变文件的当前读写位置**。

在 `deterministic_rw_verify` 方法中，使用它们的组合实现了精确控制：

1. `os.open` 以同步模式打开设备，确保 `os.pwrite` 的写入会真正落盘。
2. `os.pwrite` 根据种子生成的确定性数据，精确地写入到指定的 LBA（逻辑块地址）。
3. `os.pread` 从完全相同的位置读取数据回来进行比对，由于使用了 `pread`，即使程序在多线程环境运行，读取位置也不会错乱。

总结来说，这三个函数为测试脚本提供了**绕过所有用户态和内核态缓存、进行精确位置、原子性读写**的能力，这是验证存储设备数据完整性的基石。

- **`O_RDWR`**
  - **英文全称**：**O**pen for **R**ead an**d** **W**rite
  - **含义**：以可读和可写两种模式打开文件。其中的 `O_` 是早期 Unix 中用于“open”相关标志的通用前缀。
- **`O_SYNC`**
  - **英文全称**：**O**pen with **SYNC**hronous writes
  - **含义**：以同步写入模式打开文件。这里的 `SYNC` 是 **synchronization（同步）** 的缩写，表示写入操作会同步等待数据落盘。
- **`pread`**
  - **英文全称**：**P**ositional **read**
  - **含义**：在指定位置（position）进行读取操作。这个名字很形象地表达了它的核心功能：在读取时需要明确指定一个位置（offset），它结合了 `lseek` 和 `read` 的功能，但又是一个原子操作。
- **`pwrite`**
  - **英文全称**：**P**ositional **write**
  - **含义**：在指定位置（position）进行写入操作。与 `pread` 对应，它在写入时也需要明确指定一个位置。

这段代码实现了一个**确定性、分块的数据写入循环**，是 `deterministic_rw_verify` 方法的核心部分。我们逐行分析它的逻辑：

## 1. 初始化随机数生成器

python

```
write_rng = random.Random(seed)
```



- **作用**：用传入的 `seed` 创建一个独立的随机数生成器实例
- **关键点**：`random.Random(seed)` 创建的是一个**可独立控制**的随机数生成器，不会影响全局的 `random` 模块状态
- **确定性**：相同的 `seed` 会在每次运行时生成完全相同的随机数序列

## 2. 进入写入循环

python

```
while bytes_written < length_bytes:
```



- **循环条件**：只要已写入的字节数小于总长度，就继续写入
- **状态跟踪**：`bytes_written` 记录当前已成功写入的字节数

## 3. 计算本次写入的块大小

python

```
current_size = min(chunk_bytes, length_bytes - bytes_written)
```



- **目的**：确定这一次循环要写入多少字节
- **`min()` 的作用**：保证最后一块不会超出总长度
- **示例**：
  - 如果 `chunk_bytes=4096`，剩余 `5000` 字节 → 本次写入 `4096` 字节
  - 如果剩余 `2000` 字节 → 本次写入 `2000` 字节（最后一块）

## 4. 计算本次写入的偏移量

python

```
current_offset = offset_bytes + bytes_written
```



- **目的**：确定这次写入应该从设备的哪个位置开始
- **递进关系**：每次循环的偏移量都在前一次的基础上增加已写入的字节数
- **示例**：
  - 第一次：`offset_bytes + 0`
  - 第二次：`offset_bytes + 4096`
  - 第三次：`offset_bytes + 8192`

## 5. 生成确定性随机数据

python

```
payload = write_rng.randbytes(current_size)
```



- **`randbytes(n)`**：生成 `n` 字节的随机数据
- **确定性**：由于 `write_rng` 是用固定种子初始化的，每次调用 `randbytes` 都会产生**可预测**的字节序列
- **数据模式**：这比全 0 或全 1 更接近真实数据的特征

## 6. 写入设备

python

```
os.pwrite(fd, payload, current_offset)
```



- **`pwrite`**：带偏移量的写入，直接操作底层文件描述符
- **绕过缓存**：因为打开文件时用了 `O_SYNC`，这次写入会等待数据真正落盘才返回
- **精确位置**：写入到 `current_offset` 指定的位置，不会影响文件的读写指针

## 7. 更新哈希

python

```
sha_written.update(payload)
```



- **作用**：将刚写入的这批数据添加到 SHA-256 哈希计算中
- **目的**：最终得到一个代表**所有写入数据**的唯一指纹
- **好处**：不需要保存所有原始数据，只需要 32 字节的哈希值就能验证完整性

## 8. 更新已写入计数

python

```
bytes_written += current_size
```



- **作用**：记录已经完成了多少字节
- **循环控制**：这个累加确保循环最终会结束（当 `bytes_written` 达到 `length_bytes`）

## 整体逻辑图示

text

```
起始状态: bytes_written = 0
         total_length = 1MB
         chunk = 256KB

第1次循环: 写入 0-256KB → bytes_written = 256KB
第2次循环: 写入 256-512KB → bytes_written = 512KB
第3次循环: 写入 512-768KB → bytes_written = 768KB
第4次循环: 写入 768KB-1MB → bytes_written = 1MB (结束)
```



## 为什么这样设计？

1. **避免内存爆炸**：如果一次写入整个 1GB 区域，内存会不够用
2. **进度可控**：可以实时看到写入进度
3. **错误定位**：如果写入过程中出错，可以知道是在哪个偏移量
4. **哈希计算**：可以边写边计算哈希，不需要第二次遍历数据

这个循环实现了**将一大块数据分割成小块，用确定性随机模式依次写入，同时计算整体数据的哈希值**的功能，为后续的读取验证奠定了基础。
